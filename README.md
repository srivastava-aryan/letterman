# Message Agent v2

Generates a unique message with Gemini, emails it to *you* for approval, and only sends it
to the target person once you click Approve. Scheduling is done externally so it works even
though Render's free tier puts the service to sleep.

## What changed from v1

- Messages are generated by Gemini instead of picked from a fixed array.
- Nothing goes out to the recipient until you approve it, by email.
- `node-cron` is removed. A cron running inside a sleeping instance never fires, so
  scheduling now happens via an external service that calls your `/trigger` endpoint.
  Hitting that endpoint also wakes Render up, so it solves both problems at once.
- Pending messages are stored in MongoDB instead of memory, since the process can restart
  between "message generated" and "you clicking approve".

## Local setup

```bash
npm install
cp .env.example .env
# fill in .env
npm start
```

Test it locally:

```bash
curl -X POST "http://localhost:3000/trigger?secret=YOUR_TRIGGER_SECRET"
```

You should get an approval email at `EMAIL_USER` with Approve/Reject buttons.

## Getting the pieces you need

- **Gmail App Password**: Google Account → Security → 2-Step Verification → App Passwords.
  Regular Gmail passwords won't work with SMTP.
- **Gemini API key**: https://aistudio.google.com/apikey (free tier).
- **MongoDB Atlas**: free M0 cluster at https://www.mongodb.com/cloud/atlas — grab the
  connection string, whitelist `0.0.0.0/0` (or Render's IPs) in Network Access.

## Deploying to Render

1. Push this folder to a GitHub repo.
2. On Render: New → Web Service → connect the repo.
3. Build command: `npm install`. Start command: `npm start`.
4. Add all the variables from `.env.example` under Environment, using your real values.
   Set `BASE_URL` to the `.onrender.com` URL Render gives you (you'll know it before first
   deploy finishes — Render assigns it based on the service name you choose).
5. Deploy.

## Scheduling triggers (replaces node-cron)

Render's free web services sleep after ~15 minutes of no inbound HTTP traffic. An internal
`setInterval` self-ping does **not** prevent this, because Render's sleep policy is based on
external inbound requests, not internal process activity. So scheduling needs to live outside
the app.

### Option A: cron-job.org (easiest)

1. Create a free account at https://cron-job.org.
2. Create a job:
   - URL: `https://your-app.onrender.com/trigger?secret=YOUR_TRIGGER_SECRET`
   - Method: `POST`
   - Schedule: whatever cadence you want (e.g. daily at 8:00 AM IST, or Mon/Wed/Fri at
     10:00 / 14:00 / 20:00 — same patterns as your old node-cron schedules).
3. Repeat for each schedule you want (you can add multiple jobs, same as your old
   Option 1–4 blocks).

First request after sleep will be slow (Render cold start, 20-50s) — that's fine, cron-job.org
just waits for the response.

### Option B: GitHub Actions (if you want it versioned with your code)

Add `.github/workflows/trigger.yml` in your repo:

```yaml
name: Trigger love message
on:
  schedule:
    - cron: '30 2 * * *'  # 8:00 AM IST = 02:30 UTC
  workflow_dispatch: {}    # lets you trigger manually from GitHub UI too

jobs:
  trigger:
    runs-on: ubuntu-latest
    steps:
      - name: Call trigger endpoint
        run: curl -X POST "https://your-app.onrender.com/trigger?secret=${{ secrets.TRIGGER_SECRET }}"
```

Add `TRIGGER_SECRET` as a repo secret under Settings → Secrets and variables → Actions.
GitHub cron times are UTC and can be delayed by a few minutes under load, which is fine here.

## Manually triggering a message

Anytime you want, hit the endpoint yourself:

```bash
curl -X POST "https://your-app.onrender.com/trigger?secret=YOUR_TRIGGER_SECRET"
```

## Security notes

- `TRIGGER_SECRET` guards `/trigger` since the endpoint is public — anyone with the URL and
  secret could make it generate and email you an approval request. Keep it long and private.
- `/approve` and `/reject` require the per-message `token` stored alongside the record, so
  someone would need both the message id and its token (sent only to you) to act on it.
- Never commit your real `.env` file. Only `.env.example` should be in git.
